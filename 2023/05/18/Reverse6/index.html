逆向工程

1.实验目的

​ 重打包（repackaging）攻击是 Android 设备上一种非常常见的攻击类型。在此类攻击中，攻击者修改 从应用市场下载的流行 app，对该 app 进行逆向工程，添加一些恶意负载，然后将修改后的 app 上传到应 用市场。用户很容易被欺骗，因为很难注意到修改后的 app 和原始 app 之间的区别。一旦安装了修改后的 app，里面的恶意代码就可以进行攻击，通常是在后台。例如，2011 年 3 月，发现 DroidDream 木马已被嵌 入 Android 官方市场的 50 多个 app 中，并感染了众多用户。DroidDream 木马利用 Android 中的漏洞获取 设备的 root 访问权限。 本实验的学习目标是让学生获得 Android 重新打包攻击的第一手经验，以便他们更好地了解与 Android 系统相关的这种特殊风险，并在将 app 下载到他们的设备时更加谨慎，尤其是从那些不受信任的第三方市场。在本实验中，学生将被要求对选定的 app 进行简单的重新打包攻击，并仅在我们提供的 Android VM 上演示该攻击。应警告学生不要将重新打包的 app 提交给任何市场，否则将面临法律后果。他们也不应该 在自己的 Android 设备上运行攻击，因为这可能会造成真正的损害。

2.1环境配置

2.2 Task 1:Obtain An Android APP and Install it

2.3 Task 2: Disassemble Android App

​ 要完成重打包的攻击，需要将apk的代码进行反汇编到我们能够理解的版本Smali。这里我们使用apktools来进行操作，这里可以看到反汇编的过程。

​ APK文件只是一个压缩文件，其中包含classes.dex（编译的java源代码，称为Dalvik bytecode）resources.arsc（资源文件）、AndroidManifest.xml等。APKTool基本上是解压缩APK文件，并对其内容进行解码。对于资源文件，不需要做太多。对于Dalvikbytecode classes.dex，它被分解成smali代码，APKTool将输出的文件放入一个创建的文件夹中，其名称与APK文件的名称相同。典型的APK文件的文件夹结构反汇编后的典型文件夹结构如图2所示。该文件夹包含XML资源文件，AndroidManifest文件、源代码文件等。XML资源和AndroidManifest文件应该是可读的，而且通常是非常接近其原始形式的。非常接近于它们的原始形式。反汇编的smali代码放在smali文件夹中。通常情况下，一个smali文件包含一个Java类的代码。

2.4 Task 3: Inject Malicious Code

​ 这个任务中，我们将向目标APP注入恶意代码，这里我们选择向Smali文件中注入一个新的成分。在Android应用程序中，有四种类型的组件，活动、服务、广播接收器和内容提供者。前两种组件是应用程序最常使用的，而另外两种则不那么常见。我们可以在目标应用程序中添加这些组件中的任何一个，但攻击的最重要问题是找到一种方法来触发这些组件。最简单的是广播接收器，它可以被系统发送的广播所触发。如，当系统时间被设置时，一个TIME SET广播将被发出；系统重启后，将发出BOOT COMPLETED广播。我们可以写一个广播，我们可以写一个广播接收器来监听这些广播中的一个，这样恶意代码就会被这些事件自动触发。
​ 这里我们使用实验手册所提供的恶意代码

​ 上述恶意代码如果被触发，将删除设备上的所有联系人记录。该代码实现了一个广播接收器，它将被一个广播事件所触发。一旦它被触发，它就会进入onReceive()方法，这就是我们实现恶意逻辑的地方。上面的代码基本上与联系人应用程序的内容提供商进行交互，并要求内容提供商删除其所有条目，本质上是抹去所有的联系人记录。在代码中，一个ContentResolver被用来访问存储在手机上的联系人。为了获得联系人列表，一个查询被发送到联系人的内容提供者。这个查询没有提供任何匹配标准，所以所有的记录都通过一个Cursor对象返回。然后，代码会浏览所有这些记录，并逐一删除它们。

2.7 Task 6: Using Repackaging Attack to Track Victim’s Location

Step 1. Setting up mock locations.

​ 在安卓设备中，我们可以从GPS硬件来获取地址，但是我们的安卓模拟机没有GPS硬件，因此我们使用app自动生成位置，进行随机的选择。

Step 2: Configuring DNS.

​ 在重打包程序中的恶意代码将会把受害者的位置送到攻击者的服务器，我们将使用我们的SEEDUbuntu来做这一件事情，然后我们需要将用户名连接到它的IP地址，因此需要来修改android模拟器的host文件。

​ 然后更改xml文件

Step 4: Enabling the permission on the Android VM.

​ 我们在安卓模拟器上将APP的权限设置，打开对位置权限的读取

Step 5: Triggering the attacking code

​ 这一步的实施实际上是有一些问题的，我们在20.04上其实是没有/var/www/RepackagingAttack这个网页的部署的，因此APP会向服务器发送大量无效的连接请求，而无法有效发送位置信息，因此只能在16.04上进行实验

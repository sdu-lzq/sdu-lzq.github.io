逆向工程

1 实验目的

​ 在lab4中，我们学习了几种能够修改和增强二进制程序分析的技术，这些技术虽然使用起来相对简单，但在代码插入的数量和位置这两个方面存在一定的局限性。在本次实验中，将使用一种二进制插桩的技术，该技术能够在二进制程序的任何位置插入几乎无限的代码，以观察或修改该二进制程序的行为。

​ 本次实验中的工具：

  • Pin(Intel推出的动态二进制插桩工具)
  • IDA

2.2 Task 2：Profiling with Pin

​ 在软件工程中，性能分析（performance analysis 也称为 profiling），是以收集程序运行时信息为手段研究程序行为的分析方法，是一种动态程序分析的方法。 性能分析的测量对象是程序的空间或时间复杂度、特定指令的使用情形、函数调用的频率及执行时间 等。性能分析的目的在于决定程序的哪个部分应该被优化，从而提高程序的速度或者内存使用效率。

2.2.1 从起始处分析应用程序

​ 首先编译生成profiler.so，对/bin/true 执行分析。

​ 可以看到，这个程序总共只执行了95条指令。Profiler 以【目标地 址 <-源地址：计数值（target<-source: count）】的格式输出控制转移信息，其中计数值表示特定分 支边的执行频率，以及该分支占所有控制转移计数的百分比。

2.2.2 将Profiler附加到运行中的进程

​ 将pin附加到运行中的进程与使用pin从起始处插桩应用程序的操作类似。但是选项有一点不同。在一些 Linux 操作系统上（包括 Ubuntu 发行版）有种安全机制可以阻止 Pin 附加到运行中的进程。所 以要让 Pin 正常附加到进程上，必须暂时禁用该安全机制，（在下次重启时系统将自动重新启用该机制）。

​ 此外，还需要一个合适的测试进程来使 Pin 附加上去。为此启动了一个后台 netcat 进程，它在本地主 机上监听 UDP 的 9999 端口。通过进程的 PID 来使 Pin 附加到进程，可以在启动进程时记下或通过 ps 命 令查找获取 PID。

​ 启动 pin，-pid 选项告诉 Pin 根据给定的 PID 附加到运行中的进程。为使 netcat 监听进程执行一些指 令，而不只是阻塞等待网络输入，使用另一个 netcat 进程向监听进程发送消息“Testing the profiler”。然 后使用 fg 命令把 netcat 监听进程带到前台，并终止该进程。当应用程序终止时，Profiler 调用 fini 函 数并输出统计信息以便分析，其中包括控制转移、调用函数及系统调用的列表。

​ 下面是将pin程序附加到netcat进程上，并且我们执行并发送相关的信息，最后将后台运行的netcat调用到前台并终止，这时候pin程序也会终止并输出相关信息。

​ 从下面我们可以看到Profiler输出的具体信息，包括控制转移、调用函数及系统调用的列表。

Q：根据 Pin 执行结果，使用了哪些系统调用？

​ 我们可以从/usr/include/asm中的头文件来查看系统调用号，这里是64bit的程序，可以看到进程使用了sys_read，sys_write,sys_poll,sys_connect,sys_accept。

​ read和write进行读写操作，poll可以监视多个文件描述符，一旦某个文件描述符就绪（一般是读就绪或者写就绪），就能够通知应用程序进行相应的读写操作。因为我们这里使用了netcat，因此需要使用网络的连接请求connect和接收请求accept。

3.3 Task 3：Automatic Binary Unpacking with Pin

​ 在这个例子中，我们将看到如何使用Pin构建一个可以自动对加壳的二进制文件进行脱壳的Pintool

3.4.1 Introduction to Executable Packers

​ 可执行文件加壳器（简称加壳器）将二进制文件作为输入并将二进制代码和数据段“打包”到一个压缩 或加密的数据区域中，然后生成一个新的、加壳的可执行文件。最初，加壳器主要用于压缩二进制文件， 但现在它主要被恶意软件用来生成逆向工程师难以进行静态分析的二进制文件。

​ 通用脱壳器依赖于加壳器常见的运行模式来检测指向OEP的跳转，然后将OEP的内存区域转储到磁盘。

3.4.2 测试脱壳器

​ 现在让我们对一个用 UPX 加壳的可执行文件进行脱壳来测试一下脱壳器，下面演示对二进制文件加壳的过程。

​ 下面将使用IDA查看二进制文件来确认它已经被加壳，由于直接在/bin/ls下进行逆向分析会出现permission denied的错误，因此我们将ls源文件首先copy一份到chapter9文件夹下面。

​ 首先对加壳后的文件进行反汇编分析，可以看到二进制文件只包含4个函数，而且函数之间的调用关系比较简单

​ 而对原始的二进制文件进行分析，可以看到函数非常多，二进制文件的入口start，虽然start进去很简单，但是看到main函数调用关系很复杂

​ 现在外面对脱壳器恢复ls原始代码和数据的能力进行测试

​ 可以看到目录列表包含多个已脱壳的文件，这些文件都使用统一的命名方案，表示转储区域的起始地址、结束地址及插桩代码检测到的入口点地址。

​ 日志文件 unpacker.log 详细记录了被提取的区域信息，并列出了脱壳器找到的所有内存块（甚至包含 未脱壳的内存块）。通过 file 命令可以知道最大的脱壳文件（根据 unpacker.log 中记录确定或查看文件 大小）是一个在某种意义上有点“损坏”的 ELF 二进制文件，因为 ELF 二进制文件的内存表示并不直接对应 file 等程序所期望的磁盘表示。由于节头表在运行时不可用，因此脱壳器无法恢复它。不过，让我们看 看 IDA 和其他工具是否可以解析脱壳后的文件。

(1)可以使用strings命令看到脱壳后的二进制文件中包含许多可读的字符串

（2）使用IDA能够在脱壳的二进制文件中找到更多的函数

（3）使用objdump命令来将脱壳后的代码与ls的原始代码进行比较来作为最后的健全性检查

​ 可以正常使用 objdump 命令来反汇编原始二进制文件，但是对于脱壳后的二进制文件，需要传递一 些特殊选项以告诉 objdump 命令将文件视为包含 x86-64 代码的原始二进制文件并反汇编所有的文件内容 （用-D 选项代替通常的-d 选项）。这是因为脱壳后的二进制文件不包含可以让 objdump 命令用以确定代码 节位置的节头表。

​

​ 使用命令对脱壳后的文件进行反汇编，可以看到反汇编成功

​ 和原始文件的反汇编代码进行比较，可以看到除了代码地址不同外，二者其余的代码是相同的。地 址不同是因为 objdump 命令缺少节头表而不知道脱壳的二进制文件的预期加载地址。（上图为原始代码反汇编，下面为脱壳代码反汇编）
